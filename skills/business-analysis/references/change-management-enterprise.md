# Change Management Enterprise
> "Method over hacks."
## TL;DR
Communication plans, stakeholder alignment, rollout strategy for enterprise change. Scope: people-side adoption of a defined change — NOT the technical delivery, governance board, or project schedule (those belong to their own skills). [EXPLICIT]
## Procedure
### Step 1: Discover
- Map stakeholders by influence x impact; capture current state, change drivers, and the WIIFM per segment [EXPLICIT]
- Surface adoption barriers early: skill gaps, sponsor absence, change fatigue [EXPLICIT]
### Step 2: Analyze
- Evaluate options per Constitution XIII/XIV [EXPLICIT]
- Choose rollout shape with a JUSTIFIED trade-off:
  - Big-bang — fastest cutover, highest blast radius if adoption stalls; use only with strong sponsor + reversible change [EXPLICIT]
  - Phased/pilot — slower, but de-risks via early feedback and reference adopters; default for irreversible or high-resistance change [EXPLICIT]
### Step 3: Execute
- Implement comms plan with evidence tags; assign a named sponsor and per-segment channel/cadence [EXPLICIT]
- Run reinforcement loops (training, champions, feedback intake) — comms is repeated, not one-shot [EXPLICIT]
### Step 4: Validate
- Verify quality criteria met against adoption signals, not delivery completion [EXPLICIT]
## Quality Criteria (acceptance)
- [ ] Evidence tags applied to every non-trivial claim [EXPLICIT]
- [ ] Constitution-compliant (XIII/XIV) [EXPLICIT]
- [ ] Actionable output: every recommendation has an owner + cadence [EXPLICIT]
- [ ] Named sponsor identified and committed [EXPLICIT]
- [ ] Each stakeholder segment has a tailored message + channel [EXPLICIT]
- [ ] A measurable adoption signal is defined (e.g. usage %, ticket-volume delta) [EXPLICIT]

## Assumptions & Limits

- Assumes access to project artifacts (code, docs, configs) and an identifiable sponsor [EXPLICIT]
- Requires English-language output unless otherwise specified [EXPLICIT]
- Does not replace domain expert judgment for final decisions [EXPLICIT]
- Anti-scope: not a project plan, RACI, or steering-committee charter — redirect those [EXPLICIT]
- Produces a plan, not the change itself; adoption is observed post-rollout [EXPLICIT]

## Edge Cases & Failure Modes

| Scenario | Handling |
|----------|----------|
| Empty or minimal input | Request clarification before proceeding |
| Conflicting requirements | Flag conflicts explicitly, propose resolution |
| Out-of-scope request | Redirect to appropriate skill or escalate |
| No identified sponsor | Halt — flag absent sponsorship as the top adoption risk before any comms work [EXPLICIT] |
| Active change fatigue / prior failed rollout | Slow cadence, lead with WIIFM, surface past failure explicitly [EXPLICIT] |
| Mandated date with high resistance | State the adoption-vs-speed trade-off; recommend pilot to de-risk [EXPLICIT] |
| Comms treated as one-time | Reject — require reinforcement loop, not a single announcement [EXPLICIT] |
