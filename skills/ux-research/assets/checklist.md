# Route-selection & gate checklist — ux-research

A fast operator checklist the support/guardian agents run. Pairs with
`assets/quality-rubric.json` (machine-readable form). [DOC]

## A. Pick the route (answer the first that fits)
- [ ] Do we need to learn **why / what users need**? → `user-research`
- [ ] Do we need to measure **how many / how often** an attitude, at scale, with
      reachable n ≥ ~30/segment? → `survey-design`
- [ ] Do we have a **built or clickable flow** and need to know if users can
      complete the task? → `user-testing`
- [ ] Is **nothing built** and we need to falsify a concept first? → `prototyping`
- [ ] Two routes plausible? Ask ONE question; never run two. [INFERENCIA]

## B. Before fielding / building
- [ ] Decision the work informs is named (not "nice to know")
- [ ] Segment + recruitment source identified; internal staff excluded
- [ ] Survey only if reachable n ≥ ~30/segment and the *why* is already known
- [ ] No reachable users → outputs marked *provisional* + validation study named

## C. Before "done" (gate)
- [ ] Exactly one playbook read and followed
- [ ] Route-specific rubric criteria met (see quality-rubric.json)
- [ ] n / response rate / MoE (survey) or severity + verbatim (testing) present
- [ ] Each `[SUPUESTO]` paired with a verification step
- [ ] One tag family; no green-as-success; no PII; single brand
- [ ] Explicit, owned next step present
