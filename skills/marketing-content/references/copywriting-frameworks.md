# Copywriting Frameworks

> "Method over hacks. Evidence over assumption."

## TL;DR

Pick a proven framework (AIDA, PAS, FAB, 4Ps, BAB) to structure marketing copy and landing pages, then validate against MetodologIA brand standards and evidence tagging. Default to **PAS for cold traffic** (pain-aware) and **FAB for product pages** (feature-rich). [EXPLICIT]

## The Frameworks

| Framework | Structure | Best for | Pick when |
|-----------|-----------|----------|-----------|
| **AIDA** | Attention → Interest → Desire → Action | Ads, emails, long-form sales pages | Audience is scrolling and unaware [EXPLICIT] |
| **PAS** | Problem → Agitate → Solve | Cold traffic, pain-driven offers | A sharp, nameable pain exists [EXPLICIT] |
| **FAB** | Feature → Advantage → Benefit | Product/spec pages, B2B | Buyer compares concrete features [EXPLICIT] |
| **4Ps** | Promise → Picture → Proof → Push | Landing pages, VSL scripts | You have proof assets (data, testimonials) [EXPLICIT] |
| **BAB** | Before → After → Bridge | Transformation/outcome offers | The product changes a state [EXPLICIT] |

**Decision rule:** unaware audience → AIDA; pain-aware → PAS; comparison shopper → FAB; proof-rich → 4Ps; outcome-led → BAB. [INFERENCIA]

## Procedure

### Step 1: Discover
- Capture audience awareness stage, the single desired action (CTA), and the core offer.
- Inventory available proof: metrics, testimonials, guarantees, logos. No proof → BAB/PAS over 4Ps.
- Read existing brand voice + prior copy for tone continuity (references/brand/).

### Step 2: Analyze
- Evaluate framework fit against Constitution principles (XIII Think First, XIV Simple First).
- Select via the decision rule above; document why the runner-up was rejected. [EXPLICIT]
- One framework per asset. Do not stack (AIDA inside PAS dilutes both). [INFERENCIA]

### Step 3: Execute
- Draft to the chosen structure, one idea per beat. Lead with the benefit, not the feature.
- Map every claim to evidence; convert features to benefits ("256-bit → your data stays private").
- Apply brand template for HTML outputs (references/brand/). One CTA, repeated, never competing CTAs.

### Step 4: Validate
- Read aloud; cut any sentence that does not move toward the CTA.
- Check evidence-tag coverage and Constitution compliance.
- Confirm the framework's final beat (Action/Solve/Benefit/Push/Bridge) names the CTA explicitly.

## Worked Example (PAS)

Offer: async standup tool for distributed teams. [EXPLICIT]
- **Problem:** "Your 9am standup wakes up three timezones."
- **Agitate:** "Every sync meeting is 6 engineers waiting on 1 to finish talking — multiply by 250 workdays."
- **Solve:** "Post updates async. Reclaim the hour. → Start free."

FAB rewrite of the same product: *Feature* "Threaded async updates" → *Advantage* "No scheduled call" → *Benefit* "Engineers ship instead of sit in meetings." [EXPLICIT]

## Quality Criteria

- [ ] Exactly one framework, matched to audience awareness via the decision rule
- [ ] Every claim carries an evidence tag; features expressed as benefits
- [ ] Single, repeated CTA named in the closing beat
- [ ] Follows Constitution principles; passes read-aloud cut test
- [ ] No redundancy, hype, or unverifiable superlatives

## Anti-Patterns

| Anti-Pattern | Why It's Bad | Do This Instead |
|-------------|-------------|-----------------|
| Acting without understanding | Wastes effort on wrong solution | Think First (XIII) |
| Over-engineering | Complexity without value | Simple First (XIV) |
| Missing evidence tags | Claims without basis | Tag every assertion |
| Listing features, not benefits | Reader must self-translate value | Run FAB: feature → benefit |
| Multiple competing CTAs | Splits attention, lowers conversion | One CTA, repeated |
| Hype without proof ("revolutionary") | Erodes trust, no evidence anchor | Replace adjective with a metric/testimonial |
| Stacking frameworks in one asset | Muddles structure, no clear arc | One framework per asset |
| Picking framework before audience | Structure misfits awareness stage | Run the decision rule first |

## Assumptions & Limits

- Assumes access to project artifacts (code, docs, configs) and brand voice reference. [EXPLICIT]
- Requires English-language output unless otherwise specified. [EXPLICIT]
- Frameworks structure persuasion; they do not validate the offer's truth or legality — claims still need real proof. [EXPLICIT]
- **Anti-scope:** does not cover SEO keyword research, paid-media bidding, or visual/layout design (separate skills). [EXPLICIT]
- Does not replace domain expert or legal review for regulated claims (health, finance). [EXPLICIT]

## Edge Cases

| Scenario | Handling |
|----------|----------|
| Empty or minimal input | Request audience, CTA, and offer before drafting |
| Conflicting requirements | Flag conflicts explicitly, propose resolution |
| Out-of-scope request (SEO, design, ads) | Redirect to appropriate skill or escalate |
| No proof assets available | Use BAB or PAS; avoid 4Ps (proof-dependent) |
| Multiple equally-valid CTAs requested | Force-rank to one primary; demote rest to secondary nav |
| Regulated-claim copy (health/finance) | Draft, then route to legal/compliance review before publish |
| Audience awareness unknown | Default to PAS (safest for cold) and flag the assumption |
