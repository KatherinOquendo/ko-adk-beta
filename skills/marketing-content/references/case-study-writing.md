# Case Study Writing
> "Method over hacks."

## TL;DR
Write client case studies in **Problem → Solution → Result** order, anchored by quantified metrics, a verbatim client quote, and a scannable visual layout. Every claim carries an evidence tag. [EXPLICIT]

## Structure (canonical order)
1. **Headline** — outcome + metric, ≤12 words (e.g. "Cut onboarding from 6 weeks to 4 days"). [EXPLICIT]
2. **Snapshot** — client, industry, size, the one number that matters. [EXPLICIT]
3. **Problem** — the cost of the status quo, in the client's words. [EXPLICIT]
4. **Solution** — what was done and *why that approach*; name the trade-off rejected. [INFERENCIA]
5. **Result** — before/after metrics + one verbatim quote with attribution. [EXPLICIT]
6. **CTA** — single next step. [EXPLICIT]

## Procedure
### Step 1: Discover
- Gather context: client artifacts, baseline metrics, interview transcripts, signed quote-approval. [EXPLICIT]
- If no baseline metric exists, flag it — a case study without a before-number is a testimonial, not a case study. [INFERENCIA]
### Step 2: Analyze
- Evaluate framing options per Constitution XIII/XIV; pick the single hero metric. [EXPLICIT]
- Reject vanity metrics (impressions, "engagement") in favor of business outcomes (revenue, time, cost, retention). [INFERENCIA]
### Step 3: Execute
- Draft in canonical order with evidence tags; quote verbatim, never paraphrase a client. [EXPLICIT]
### Step 4: Validate
- Verify every metric traces to a source; confirm quote approval on file before publish. [EXPLICIT]

## Acceptance Criteria
- [ ] Hero metric is a business outcome with explicit before/after [EXPLICIT]
- [ ] ≥1 verbatim, attributed, approved client quote [EXPLICIT]
- [ ] Every metric is source-traceable (no unsourced numbers) [EXPLICIT]
- [ ] Evidence tags applied; Constitution-compliant; single-brand [EXPLICIT]
- [ ] Scannable: headline + subheads readable in <30s [INFERENCIA]

## Worked Example (skeleton)
> **Headline:** "Reduced support backlog 71% in one quarter"
> **Problem:** "Tickets aged 9 days on average; churn risk rising." [client quote]
> **Solution:** Triage automation + tiered routing. Rejected full rip-and-replace (too slow). [INFERENCIA]
> **Result:** 9d → 2.6d resolution; CSAT 3.2 → 4.5. "We finally got ahead of it." — Head of Support [EXPLICIT]

## Assumptions & Limits
- Assumes access to project artifacts (code, docs, configs) and baseline metrics [EXPLICIT]
- English-language output unless otherwise specified [EXPLICIT]
- Does not replace domain expert judgment for final decisions [EXPLICIT]
- **Anti-scope:** never invent metrics, attribute unapproved quotes, name prices, or claim causation a single data point cannot support. [SUPUESTO]

## Edge Cases & Failure Modes
| Scenario | Handling |
|----------|----------|
| Empty or minimal input | Request clarification before proceeding |
| Conflicting requirements | Flag conflicts explicitly, propose resolution |
| Out-of-scope request | Redirect to appropriate skill or escalate |
| No baseline / before metric | Label as testimonial; do not publish as case study [INFERENCIA] |
| Quote not yet approved | Hold publish; use [PENDING-APPROVAL] placeholder [EXPLICIT] |
| Client requests anonymity | Anonymize to role+industry; keep metrics; drop logo [EXPLICIT] |
| Metric cannot be sourced | Cut the number rather than ship it unsourced [EXPLICIT] |
